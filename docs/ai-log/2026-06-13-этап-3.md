# AI-лог: 2026-06-13 — realtime-дуэли (этап 3)

## Процесс

Спека `docs/specs/duels.md` написана оркестратором как жёсткий контракт между тремя сервисами (формы internal REST, WS-протокол, Redis-ключи, Эло). Два параллельных Opus-субагента:
- **API-домен дуэлей** (apps/api): internal-эндпоинты создания/финиша, finish-транзакция (FOR UPDATE обеих строк ratings в порядке user_id, идемпотентность, Эло K=32), aiokafka-продюсер `duels.finished`.
- **Realtime с нуля** (apps/realtime): NestJS + Socket.IO + Redis adapter, матчмейкинг с Lua-пейрингом, движок дуэли с серверными дедлайнами, реконнект, kafkajs-продюсер `answers.submitted`.

## Что субагенты сделали хорошо сами

- API: идемпотентность finish через `duels` FOR UPDATE как арбитр гонки; строки ratings создаются заранее, тай-брейк и ничья по спеке.
- Realtime агент сам нашёл и починил read-modify-write баг в записи ответов (два одновременных ответа клоббели progress) — заменил на атомарный Lua-скрипт. Это ровно тот класс гонок, что ТЗ §11 п.10 называет «самым сложным».

## Критический баг интеграции (поймал оркестратор)

Агенты работали изолированно и **разошлись в форме эталона**: Core API отдаёт `answer` как `{"correct": <int>}` (единый формат с соло-режимом), а realtime типизировал его как `answer: number` и сравнивал `selected === task.answer`. Юнит- и e2e-тесты realtime использовали собственный мок internal-client с числовой формой, поэтому расхождение там не всплывало — **зелёные тесты обоих сервисов это не ловили**.

Поймано **живым интеграционным прогоном** (не тестами): поднял API + realtime + Redis + Redpanda, двумя реальными WS-клиентами прошёл полную дуэль. Симптом — `correctOption` приходил как `{correct:0}`, все ответы `correct:false`.

Фикс: нормализация на границе — тип `CreateDuelTask.answer = {correct: number}`, извлечение `.correct` в матчмейкинге при построении состояния (движок сохраняет допущение `answer: number`). Тестовые хелперы разделены: `makeTasks` (API-форма) и `makeDuelTasks` (нормализованная), чтобы e2e гонял реальный путь нормализации.

**Вывод для процесса:** контракт между сервисами недостаточно описать прозой в спеке — точную форму JSON-полей (`answer`) надо фиксировать примером, а интеграцию проверять живым прогоном, а не только моками. Зелёный CI обоих сервисов не доказывает их стыкуемость.

## Живая верификация (после фикса)

Полный цикл против реального стека: матчмейкинг → countdown → 3 задачи → финиш. Победа корректного игрока, Эло −16/+16 (K=32, симметрично, zero-sum) записано в БД через FOR UPDATE-транзакцию; 12 ответов в `answers`; оба Kafka-топика (`answers.submitted`, `duels.finished`) созданы, события доставлены в Redpanda. `correctOption` — число, раскрывается только автору после ответа; соперник видит лишь время и факт верности (эталон клиенту не утекает).
